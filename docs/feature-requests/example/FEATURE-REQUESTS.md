### 1. Auto-commit on a stale work tree so the loop never stops

**Original wording:**
> Currently the script will stop and return stale work tree aborting commit or stash before
> rerunning and completely stop the loop when this stale work tree is encountered. I'd like to
> add automation to automatically perform the commit for the user so that the script never stops
> continuing its loop. Determine the best way to do this, either by updating the python code, the
> PROMPT.md, or both.

**Request:**
When the runner hits a stale work tree (uncommitted changes that currently force it to abort the
commit/stash and halt the loop), have it automatically commit the outstanding work so the loop
keeps running instead of stopping.

**Why:** The loop should run unattended; a stale work tree currently breaks the AFK promise.

**Open questions for grilling:**
- Fix in the Python runner, in `PROMPT.md`, or both?
- What should the auto-generated commit message be, and should anything ever still halt the loop?

**Context:** `./context/SAMPLE-TERMINAL-OUTPUT.txt` — shows current loop output where this occurs.

### 2. Show per-iteration context-window usage and report it in the run summary

**Original wording:**
> I'd like to try to add a feature that will show context percentage usage during each individual
> iteration as a ticker that updates as context is consumed. I'd also like for this to be something
> that is shared back as output for the run summary. For each iteration, there should be a table
> column produced as output that includes how much context in the model's context window was used
> for whatever model was selected for the loop.

**Request:**
Display a live ticker of context-window usage (as a percentage) during each iteration as context is
consumed, and surface the same figure in the end-of-run summary — a per-iteration table column
reporting how much of the selected model's context window was used.

**Open questions for grilling:**
- Where does the context-usage number come from for an arbitrary selected model?
- Live ticker rendering vs. the final summary table — same source, different display?

**Context:** none

### 3. Make the terminal output more human-readable and animated

**Original wording:**
> I'd like to explore potential ways that we could improve the output in the terminal that's
> generated, so it's more human readable or even potentially more animated with loading indicators
> etc. as the loop continues until it finishes. Some things that we could do to improve the human
> readability might be additional colors indicating different activities or other indicators that
> will help the user understand what's happening other than some generic read bash, bash stop bash
> indicators as an example.

**Request:**
Improve the loop's terminal output for human readability — e.g. loading/progress indicators while
the loop runs, and colors or other cues that distinguish different activities, rather than the
current generic `read bash` / `bash` / `stop bash` indicators.

**Open questions for grilling:**
- Which activities deserve distinct colors/indicators, and what's the desired visual vocabulary?
- How much animation is wanted vs. keeping logs greppable/CI-friendly?

**Context:** `./context/SAMPLE-TERMINAL-OUTPUT.txt` — example of the current generic output.
