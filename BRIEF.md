# Features

- Merge the dashboard and summary views into a single view that we'll simply call the dashboard.
- Remove the log tab to ensure there are no tabs left. When a user selects an issue and presses enter, it should direct them to that issue's log. The log stream should automatically scroll to display the latest log output, complete with timestamps indicating when each log entry was generated.
- When viewing the log output of an issue, that view should automatically stream to the bottom of the log, continuously updating to show the latest entries. This eliminates the need for the user to manually scroll down to see the most recent log output.
- The log should include timestamps for the output it generates, as well as for activities such as when the GitHub Copilot model is thinking.
- Update the defaults so that when the user runs the `afk.sh` script, it does not ask for the user to specify the model when everything is loaded. We should make this a parameter: if the user wants to enter a model selection mode, they could pass it up as a parameter when the script runs. If they do not, we should default to opus 4.8 with max reasoning because that is what the variable paremeters in `afk.sh` are telling it to use. The user shouldnt be asked to pick a model and reasoning effort if these values are set like they currently are. A parameter to enter model selection mode should be passed in.
- On the dashboard, remove the waiting column for every single queued issue. User should see the time the script started and the time a specific issue began, and how long it took to complete an issue after it completed. The table columns and row values from the summary tab should be merged into the current dashboard table as additional columns that will popululate dynamically with their values to reflect consumption as issues are progressed until completion. For example, tokens in, tokens out, the cost as it increases, etc.
- The AFK loop sometimes quits because changes to the project, as issues are worked on, aren't being committed to GitHub as they're made. This makes the ralph loop think it's done with all its issues and/or forces it to exit because the working tree has unstaged commits or changes that need to be pushed in order for it to continue working. Therefore, we need to implement a mechanism to ensure this doesn't happen.
